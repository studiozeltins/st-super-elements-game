# Phase 3 — UI Design Contract

> Visual and interaction contract for shards-at-risk (death loss, PVP steal, erosion). Generated by gsd-ui-researcher, verified by gsd-ui-checker.
> **Scope is small and additive.** Three tiny visible surfaces, each a faithful mirror of something already shipped: (1) a `shard_drop` ground pickup that reuses the gem-drop 3D renderer with a distinct purple "rare shard" look; (2) a self-facing shard lost/stolen HUD toast reusing the existing event-driven feedback pattern (`App.tsx` pvpHit/healEvent onInsert → `gameRef`) plus a `transcendShards` diff; (3) the existing `.wallet-chip` shard counter, made to flash on loss/gain. **No new screens, no new palette, no new tokens, no new broadcast event table.** Reuse the shipped token system in `src/index.css` and the gem machinery in `src/game/createGame.ts`.

**shadcn gate:** not run and not applicable. **Overriding rule for this phase:** everything here belongs to the **transcend-shard family**, which the codebase already colors purple (`--shard #9333ea`). The gem drop is gold; the shard drop must read as a rarer purple relative of it. Reuse the existing `.wallet-chip` (shard chip, `index.css:1118`), the gem renderer (`createGemMesh`/`updateGemDrops`, `createGame.ts:610-661`), and the event-driven self-number toast pattern (`App.tsx:157-174`) as the three exemplars. The only net-new CSS is a loss-flash variant of the already-shipped `.wallet-chip--pulse` and a small self-facing toast surface; the only net-new render code is a purple variant of the gem mesh.

## Spacing Scale

The codebase does **not** use a strict 8-point scale (it mixes 10/11/12/13/15px type and 5/10/12px padding). This phase conforms to the existing exemplars rather than imposing a new scale. The two DOM surfaces (toast + counter flash) inherit their box metrics from shipped classes; the 3D drop is world-space (Three.js units, not px).

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon–number inline gap inside the chip / toast |
| sm | 8px | Toast internal padding (vertical) |
| md | 12px | Toast internal padding (horizontal), chip-to-chip gap |
| lg | 16px | Toast offset from the HUD edge / player anchor |

**Phase-specific values (locked, matching exemplars):**
- Shard counter flash: **reuses `.wallet-chip` box metrics verbatim** (`index.css:1118-1126`) — no new padding.
- Toast text metrics: reuse the floating self-number treatment already used by `spawnSelfNumber` (world-anchored over the player) — this phase adds no new px box if it routes through `gameRef` self-numbers; if a DOM toast strip is used instead, pad **8px 12px** and offset **16px** from the HUD safe area.

**World-space (Three.js units, not the px scale):**
- Shard drop mesh radius: **0.42** (1.4× the gem `GEM_RADIUS = 0.3`, `createGame.ts:274`) so a scarce shard reads visibly larger/rarer than a gem.
- Hover bob, spin, magnet radius, collect radius, pickup grace: **identical to gems** (reuse `GEM_PICKUP_DELAY = 1.2`, `GEM_MAGNET_RADIUS = 5.5`, `GEM_COLLECT_RADIUS = 1.1`, `createGame.ts:270-272`).

Exceptions: inherits existing non-4-multiple type sizes from the exemplars; do not "fix" them here.

## Color

Existing token palette (`src/index.css` `:root`, lines 17–25), mapped to the 60/30/10 split. **This phase introduces zero new color values.**

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--ink` `#0c110e` | Screen fill, canvas backdrop |
| Secondary (30%) | `--panel` `rgba(12,17,14,0.72)` + `--line` `rgba(232,245,224,0.16)` | Any toast panel backing |
| Accent — chrome (green) | `--accent` `#7ec843` | UI chrome only (tabs, focus). **NOT used by this phase.** |
| Accent — constellation (gold) | `--gold` `#f2c14e` | Gems + gem ground drops + constellation rows. **NOT used by this phase's shard surfaces.** |
| Accent — transcend (purple) | `--shard` `#9333ea` | **This phase's family color:** the `shard_drop` mesh (color + emissive), the shard counter chip + its flash, the `◈` glyph in every toast, and the toast text. Mesh hex form: `0x9333ea`. |
| Destructive | `--danger` `#ff5c3c` | Destructive user actions only. **Not used here** — losing a shard is an event that happens *to* the player, not a destructive action they confirm; keep `--danger` reserved (consistent with Phase 1/2). |

**Accent reserved-for (this phase):** `--shard` purple is reserved exclusively for the transcend-shard surfaces — the ground `shard_drop`, the shard counter chip and its loss/gain flash, and the `◈` glyph + text in the shard toasts. This preserves the shipped semantic contract: **purple = Constellation Shard** (distinct from gold gems and green chrome). Never color the `shard_drop` gold; never color a gem drop purple.

**Loss-flash coloring (locked, no `--danger`):** a shard lost/stolen is a real negative beat, but per the shipped contract `--danger` stays reserved for destructive user actions. The loss flash therefore stays **in the shard family** and is distinguished from the gain flash by **motion, not hue** — see Interaction & States (`.wallet-chip--drain`). This keeps the loss unmistakably "shard-related" while never introducing red into a non-destructive event.

## Component Contract — `shard_drop` Ground Pickup (3D)

A distinct purple "rare shard" mirror of the gold gem drop. Reuses the entire gem render/magnet/collect pipeline; only the mesh look differs.

| Property | Value (locked) |
|----------|----------------|
| Base machinery | Clone `createGemMesh` + `updateGemDrops` + `syncGemDrops` into `createShardMesh` / `updateShardDrops` / `syncShardDrops` (`createGame.ts:610-661`, `937-966`). Reuse magnet, grace gate, dedupe set, and the collect request wholesale. |
| Geometry | `THREE.OctahedronGeometry(0.42)` — same crystalline family as the gem octahedron, scaled **1.4×** (`GEM_RADIUS = 0.3` → `0.42`) so it reads as rarer/heavier. |
| Color / emissive | `--shard` purple `0x9333ea` for both `color` and `emissive`. `emissiveIntensity: 0.6` (vs gem `GEM_EMISSIVE_INTENSITY = 0.4`, `createGame.ts:275`) — a brighter self-glow signals rarity. |
| Sparkle | **Always sparkles** (reuse the rare-tier sparkle path that gems 100/500 use, `createGame.ts:635-639`), tinted `0x9333ea`. A shard is always "rare tier". |
| Landing burst | Purple burst on spawn (`effectSystem.spawnBurst(pos, 0x9333ea, 12)`) — mirror the gem landing sparkle (`createGame.ts:624`) but purple and slightly denser. |
| Motion | Identical to gems: spin `deltaSeconds * 2.4`, bob `sin(elapsed*3)*0.12`, `baseY = ground + 0.9` (may raise to `+1.0` so the larger mesh clears the ground) (`createGame.ts:631-632, 620`). |
| Pickup grace / magnet / collect | Reuse gem values verbatim: `GEM_PICKUP_DELAY = 1.2`s grace, `GEM_MAGNET_RADIUS = 5.5`, `GEM_COLLECT_RADIUS = 1.1` (`createGame.ts:270-272`). On collect, call the new `network.sendCollectShard(rawId)` bridge (mirror `sendCollectGem`, `createGame.ts:658`; wired in `App.tsx:348`). |
| Data source | Read-only from the subscribed `shard_drop` table (`tables.shardDrop`). Add to the `App.tsx` subscribe list (`App.tsx:73-90`), `const [shardDropRows] = useTable(tables.shardDrop)` (`App.tsx:106`), and `gameRef.current?.syncShardDrops(shardDropRows)` (`App.tsx:372`). Server owns whether a grab succeeds; the client only renders + requests. |
| "Rare shard, not a gem" requirement (checker will verify) | The drop must be immediately distinguishable from a gem at a glance: **purple (not gold), larger (1.4×), brighter glow, always sparkling.** Do not reuse the gem color/scale. |

## Component Contract — Shard Counter (existing `.wallet-chip`)

No new chip. The purple shard chip already exists in both CharacterScreen and Gacha (Phase 1, `.wallet-chip`, `index.css:1118`). This phase only makes it **react and flash** on change.

| Property | Value (locked) |
|----------|----------------|
| Reactive update | The counter already re-renders from the subscribed `player.transcendShards` (`useTable(player)`, threaded as a prop — Phase 1). No change needed for the number to update; it updates for free on every gain (steal / pickup) and loss (death / theft). |
| Gain flash | **Reuse the already-shipped `.wallet-chip--pulse`** (grow pulse, `index.css:1128-1134`) on any increase — same feedback Phase 2 used for a successful transcend. Toggle the class for one animation cycle when the local `transcendShards` diff is UP. |
| Loss flash | New sibling **`.wallet-chip--drain`**: reuse the `.wallet-chip--pulse` keyframe *shape* but invert the scale (a quick **shrink**, e.g. `scale(0.82)` at 35%) and dim the text-shadow, staying in `--shard` purple. Distinguishes loss from gain by **motion, not color** (no `--danger`). Toggle when the local diff is DOWN. |
| Trigger | A `useRef` holding the previous `transcendShards` + a `useEffect` diff (the same minimal pattern the research recommends). UP → `--pulse`; DOWN → `--drain`. Self-clearing after one cycle. |

## Component Contract — Shard Lost / Stolen Toast (self-facing)

A short self-facing HUD message so the player feels the stake immediately in-combat. **Reuses the existing event-driven feedback pattern** (`App.tsx:157-174`, where `pvpHit`/`healEvent` onInsert call `gameRef.current?.spawnSelfNumber(...)`). **No new broadcast `shard_steal` event table** (locked, per orchestrator + research recommendation A6).

| Property | Value (locked) |
|----------|----------------|
| Mechanism | Self-facing only. Driven by the local `transcendShards` diff (same `useRef` prev + `useEffect` as the counter flash) combined with the action the local client just performed. Routes through a `gameRef` floating self-message (extend the `spawnSelfNumber` pattern to accept a short text, or add a sibling `spawnSelfMessage`). No new table, no cross-player broadcast. |
| Disambiguation (locked) | The three cases are distinguished by the local player's role in the shard movement, all knowable client-side without a new table: **(a)** local diff DOWN + local player just died with no PVP killer → PVE drop; **(b)** local diff DOWN + a fatal `pvpHit` on me (existing subscribed `pvpHit` table, `App.tsx:157`) → stolen (victim); **(c)** local diff UP coinciding with a PVP kill I just landed → stole (killer). A plain UP diff from walking over a `shard_drop` (`collectShard`) is a **pickup, not a theft** → counter pulse only, **no toast** (prevents mislabeling pickup as steal). |
| Position | Self-anchored over the local player (same anchor as the pvp/heal self-numbers) or a fixed HUD slot near the shard chip. Non-blocking, auto-dismiss (~1.5s), never requires interaction. |
| Glyph | `◈` (U+25C8) in `--shard`, matching the chip and the drop. |
| Frequency | One toast per shard event; `SHARD_DEATH_LOSS = 1`, so copy is singular. |

### Erosion feedback (secondary, when 0 shards)

When the victim carries **0 shards**, nothing is stolen/dropped — instead a transcend level (then, last-resort, one C-level) erodes. The primary readout for this is the **CharacterScreen badge decrementing** (`C6 · T{n}` from the Phase 2 contract updates reactively; the install control's section kicker `T{n} / {MAX}` re-renders). An optional in-combat erosion toast may reuse the same self-message mechanism. This is **secondary** — the counter/badge are the source of truth; the erosion toast is a nice-to-have, not a checker-blocking element.

## Interaction & States

| State | Condition | Affordance | Copy (Latvian) |
|-------|-----------|-----------|-----------------|
| **Shard on ground (fresh)** | `shard_drop` row < 1.2s old | Purple octahedron, spinning/bobbing/sparkling; **not yet collectible** (grace). Magnet inactive. | none (world object) |
| **Shard on ground (armed)** | `shard_drop` row ≥ 1.2s old | Same look; now magnets toward a nearby local player and auto-requests `collectShard` within `GEM_COLLECT_RADIUS`. Anyone can grab it. | none |
| **Pickup (gain, not theft)** | Local player collects a ground shard | Counter `.wallet-chip--pulse`; **no toast** (pickup ≠ steal). | none |
| **Lost — PVE death** | Local player dies to PVE while carrying ≥1 shard | Shard spills as a ground `shard_drop` at the death spot; counter `.wallet-chip--drain`; self toast. | `Zvaigžņu šķemba nokrita` |
| **Stolen — victim** | Local player PVP-killed while carrying ≥1 shard | Counter `.wallet-chip--drain`; self toast. No ground drop (shard went to killer). | `Zvaigžņu šķemba nozagta!` |
| **Stole — killer** | Local player lands a PVP kill on a shard-carrier | Counter `.wallet-chip--pulse`; self toast (the "steal-feed hint"). | `Nozagi zvaigžņu šķembu!` |
| **Eroded (0 shards)** | Death with 0 shards → transcend level, then a C-level, erodes | No counter change (0→0); CharacterScreen `C6 · T{n}` badge decrements reactively; optional secondary self toast. | *(optional)* `Pārsniegšanas līmenis zaudēts` |
| **Nothing to lose** | Death with 0 shards, 0 transcend, C0 | No shard event; existing gem-spill + respawn unchanged. | none |
| **Collect race (server rejects early grab)** | Client requested a grab inside grace / after another player took it | No crash; the `shard_drop` row simply vanishes on the next push and the mesh is removed. Self-healing. | none (silent) |

**Confirmation:** none. Shard loss is involuntary (death/theft) and pickup is a single walk-over — no dialogs, consistent with the gem-collect model. **Distinctness requirement (checker will verify):** the three shard-movement copies (PVE drop / stolen-victim / stole-killer) must be textually distinct, and a ground **pickup** must NOT show the "stole" toast.

## Copywriting Contract

All copy is Latvian (the HUD is fully Latvian). The rare-currency noun shown to players is **"zvaigžņu šķembas"** ("Constellation Shards"), matching the Phase 1 shard-chip label; code stays `shard`. The glyph is `◈`.

| Element | Copy |
|---------|------|
| Primary CTA | none — this phase has no button; all events are involuntary (death/theft) or a walk-over pickup |
| Toast — lost to PVE death | `Zvaigžņu šķemba nokrita` *(A Constellation Shard dropped)* |
| Toast — stolen (victim side) | `Zvaigžņu šķemba nozagta!` *(A Constellation Shard was stolen!)* |
| Toast — stole (killer side / steal-feed hint) | `Nozagi zvaigžņu šķembu!` *(You stole a Constellation Shard!)* |
| Toast — eroded, 0 shards (optional secondary) | `Pārsniegšanas līmenis zaudēts` *(A transcendence level was lost)* |
| Ground pickup | none — counter pulse is the feedback (pickup is not a theft) |
| Empty state | none — no list/screen is added; the ground has drops or it doesn't |
| Error state | none — collect races self-heal via subscription; no failure surface to message |
| Destructive confirmation | none — shard loss is an event that happens *to* the player, not a destructive action they confirm |
| Shard drop / toast glyph | `◈` (U+25C8), `--shard` purple |
| Counter aria (unchanged) | `Zvaigžņu šķembas: {n}` (inherited from the Phase 1 chip) |

**Naming note (non-blocking):** verbs chosen for the native-Latvian voice — `nokrita` (dropped/fell), `nozagta` (was stolen, victim), `Nozagi` (you stole, killer). All reference "zvaigžņu šķembas" to stay consistent with Phase 1/2. If the planner prefers alternative phrasing, swap the strings only — mechanism/color/glyph/placement are unaffected.
