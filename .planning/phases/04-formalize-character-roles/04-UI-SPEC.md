# Phase 4 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope of this phase's UI deliverable:** exactly one new visual element — a **combat-role badge**
(`tank | dps | healer | support`) shown on the character identity area. It reuses the existing
bespoke HUD design system verbatim; no new design language is introduced (per phase constraint).
Everything else (role storage in `CHARACTER_STATS`, `characters.ts` field, `serverSync.test.ts`)
is non-visual and out of this contract's scope.

**Surfaces touched:**
- `src/ui/CharacterScreen.tsx` — badge in the `.cchar__id` identity block (full-page character screen).
- `src/ui/CharacterSheet.tsx` — badge in the `.sheet__meta` row (character card modal).

## Spacing Scale

Declared values (4-based, matching existing HUD usage):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 2px | Stacked tag/label gaps inside `.cchar__id` |
| sm | 4px | Badge-to-glyph / inline micro-gap |
| md | 8px | Gap between badges in the `.sheet__meta` row |
| lg | 16px | `.cchar__id` inset from panel edge (matches existing `left:16px`) |

**Badge box metrics (reuse the existing tag pattern verbatim — do NOT re-derive):**
- Padding: `5px 10px` (identical to `.cchar__active-tag` / `.cchar__transcend-tag`).
- Border: `1px solid currentColor`.
- Stack gap in `.cchar__id`: `margin-top: 10px` (matches the active/transcend tags it sits beside).

Exceptions: the `5px` vertical padding and `10px` stack offset are **inherited from the shipped
tag components** (`.cchar__active-tag`, `.cchar__transcend-tag`) and are kept for pixel-exact
visual parity. Do not "correct" them to 4/8 — the badge must line up with the tags already there.

## Typography

Role badge reuses the shipped tag typography exactly. Reference scale below is the HUD's
existing label/body ramp; the badge is the **Label** row.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Badge / Label | 10px | 700 (`--font-display`, Pixelify Sans) | 1 (single-line pill) |
| Body | 13px | 400 (`--font-body`, Chakra Petch) | 1.5 |
| Heading | 16px | 700 (`--font-display`) | 1.2 |
| Display / Title | 20px+ | 700 (`--font-display`) | 1.2 |

Badge glyph string (CSS shorthand, copy from existing tags): `font: 700 10px var(--font-display)`
with `letter-spacing: 0.12em` and uppercase Latvian text. This is the same declaration used by
`.cchar__active-tag` and `.cchar__transcend-tag` — the role badge must be visually a third
sibling in that family.

## Color

Reserved existing tokens (do NOT repurpose): `--accent #7ec843` = active-character identity,
`--gold #f2c14e` = gems / cieņa stars, `--shard #9333ea` = transcend/būsts, `--danger #ff5c3c`
= destructive actions only. Because those four hues are already spoken-for, the four combat
roles need their own dedicated hue channel.

**Add four new role tokens to `:root` in `src/index.css`:**

| Token | Value | Role | Rationale |
|-------|-------|------|-----------|
| `--role-tank` | `#4a8fd8` | tank | Steel/defence blue — reads as "shield", distinct from every reserved hue |
| `--role-dps` | `#f0476b` | dps | Rose-crimson — "damage" without colliding with the warmer `--danger` orange-red |
| `--role-healer` | `#2fd8a6` | healer | Mint-teal — "restore", clearly separated from the lime `--accent` green |
| `--role-support` | `#7c8cf8` | support | Periwinkle indigo — "utility/aura", bluer than the magenta `--shard` purple |

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--ink #0c110e` | App background / character-screen surface |
| Secondary (30%) | `--panel rgba(12,17,14,0.72)` | Cards, rails, roster chips, tab panels |
| Accent (10%) | role tokens above (+ existing `--accent`/`--gold`/`--shard`) | Small badges, active markers, chips |
| Destructive | `--danger #ff5c3c` | Destructive actions only — NOT used by any role |

**Accent reserved for:** the role badge is a text pill with `color: var(--role-*)` and
`border: 1px solid currentColor` (colored outline + colored uppercase text on the dark panel —
no filled background). This mirrors how `.cchar__active-tag` (green) and `.cchar__transcend-tag`
(purple) already render, so tank/dps/healer/support read as four more facts in the same visual
grammar. Role color is **never the only channel**: every badge also carries a Latvian text label
(see Copywriting), so it is legible to color-blind players and satisfies the non-color-alone rule.

Theming note: role tokens live in `:root` as global defaults. The five HUD skins
(`frost/synth/verdant/abyss/celestia`) may optionally re-map `--role-*` later; they are not
required to for this phase.

## Copywriting Contract

The badge is **read-only / non-interactive** — this phase adds no button or CTA. The "copy" that
matters is the four fixed role labels (uppercase, Latvian, matching the app's language) and their
accessible names.

| Role value | Badge label (uppercase) | `aria-label` | Meaning |
|------------|-------------------------|--------------|---------|
| `tank` | `SARGS` | `Loma: Sargs` | High-HP frontliner (guardian) |
| `dps` | `UZBRUCĒJS` | `Loma: Uzbrucējs` | Primary damage dealer |
| `healer` | `DZIEDNIEKS` | `Loma: Dziednieks` | Party healer |
| `support` | `ATBALSTS` | `Loma: Atbalsts` | Utility / crowd-control (anemo) |

| Element | Copy |
|---------|------|
| Primary CTA | Not applicable — the badge is a passive indicator, no action in this phase |
| Empty state heading | Not applicable — every character always has exactly one seeded role (Success Criterion 1); the badge is never empty |
| Empty state body | Not applicable |
| Error state | Defensive fallback only: if a character's `role` is ever missing/invalid, render **no badge** (omit it silently) rather than a broken/empty pill — never show a placeholder or error string to the player |
| Destructive confirmation | Not applicable — no destructive action in this phase |

## Component Contract — Role Badge

- **New class:** `.cchar__role-tag` (character screen) and `.sheet__role` (character card). Both
  clone the `.cchar__active-tag` box model (`padding:5px 10px; font:700 10px var(--font-display);
  letter-spacing:0.12em; border:1px solid currentColor;`). Color set per role via a modifier or
  inline `color: var(--role-*)`.
- **Role → label/color mapping** is derived once from the `role` field (from `characters.ts`);
  do not hard-code per-character. A small map `ROLE_META: Record<Role, {label; token}>` keeps the
  four labels + tokens in one place shared by both surfaces.
- **CharacterScreen placement:** inside `.cchar__id`, in the tag stack next to
  `.cchar__active-tag` / `.cchar__transcend-tag` (top-left over the portrait). Order:
  active-tag → role-tag → transcend-tag (role always shown; the other two are conditional).
- **CharacterSheet placement:** append to the `.sheet__meta` row (alongside stars / weapon /
  `C{n}` level chip) as a fourth inline item, `8px` gap.
- **State:** exactly one badge per character, always visible (owned or not). Non-interactive
  (no hover/press/focus states — it is not a button; use a `<span>`, not `<button>`).
- **Accessibility:** `role` text label + `aria-label="Loma: {Label}"`; color is redundant, not
  required for comprehension.
