# Phase 1 — UI Design Contract

> Visual and interaction contract for the transcend-shard balance readout. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
> **Scope is small and additive:** surface the player's `transcendShards` balance in two existing screens (`GachaScreen.tsx`, `CharacterScreen.tsx`) by mirroring the existing gems wallet chip. No new screens, no redesign.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (bespoke CSS token system in `src/index.css`) |
| Preset | not applicable |
| Component library | none (hand-written React + CSS HUD over a pixel-art canvas) |

**shadcn gate:** not run. This is a live custom-CSS game HUD; introducing shadcn is out of scope and would break visual consistency. Registry safety: not applicable (no registries). Confirmed no `components.json` / `tailwind.config.*` present.

**Overriding rule for this phase:** match the existing gems wallet chip exactly. Do not invent new tokens; reuse the values below verbatim.

## Spacing Scale

The codebase does **not** use a strict 8-point scale (it mixes 11/14/15px type and 14/24/40/48px padding). This phase conforms to the existing wallet treatment rather than imposing a new scale. Reference scale for any net-new gap:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon–number inline gap |
| sm | 8px | Compact element spacing |
| md | 16px | Default element spacing |
| lg | 24px | Section padding |

**Phase-specific additions (locked):**
- Glyph→number inline gap inside a chip: reuse the existing pattern — a single space with the chip's `letter-spacing: 0.04em` (no new rule needed).
- Gap between the shard chip and the gems/owned readout when both sit in the same wallet row: **12px** (multiple of 4).

Exceptions: this phase inherits existing non-4-multiple values (e.g. `.gacha__wallet` `font-size: 15px`) for consistency with the exemplar; do not "fix" them here.

## Component Contract — Transcend Shard Chip

The single UI element this phase adds. One consistent pattern used in **both** screens.

| Property | Value (locked) |
|----------|----------------|
| Glyph | `◈` (U+25C8) — a crystalline "shard", visually distinct from the gems star `✦` and the owned-count `❖`. |
| Color | `--gold` `#f2c14e` (identical to gems chip) |
| Size / type | 15px, Chakra Petch, `letter-spacing: 0.04em` (identical to `.gacha__wallet`) |
| Content | `◈ {transcendShards}` — glyph then integer count |
| Number format | Raw integer, no thousands separator, matching the existing `{gems}` render. Shards are scarce (typically single/low double digits). |
| Zero / empty state | `0` renders plainly. The chip is **always shown** even at zero (no hidden/empty variant, no placeholder copy). |
| Data source | Read-only from the subscribed `player` row: `transcendShards={myPlayer?.transcendShards ?? 0}` threaded through `App.tsx` as a `number` prop. Client never computes or mutates it. |
| Accessible name | `aria-label="Zvaigžņu šķembas: {n}"` (Latvian, ties to the existing `ZVAIGZNES` constellation theme). Optional matching `title` tooltip. |

**Recommended DRY approach (non-blocking):** extract the shared visual treatment into one class (e.g. `.wallet-chip`, gold + 15px + `letter-spacing: 0.04em`) that both the existing gems readout and the new shard readout consume. Acceptable alternative: a screen-local class per screen that copies the `.gacha__wallet` values. Either satisfies the contract; the shared class is preferred.

### Placement

**GachaScreen** (`src/ui/GachaScreen.tsx`, wallet at lines ~218–229):
- Render the shard chip **inside `.gacha__wallet`, immediately to the left of** the existing gem/owned-count readout, separated by the 12px gap.
- Show the shard chip **on every tab** (it is a persistent currency, relevant regardless of the characters/banners/party tab), even though the adjacent readout swaps between owned-count (`❖ n/total`) and gems (`✦ n`).
- Result on a non-characters tab: `◈ {shards}   ✦ {gems}` (shards first, gems second).

**CharacterScreen** (`src/ui/CharacterScreen.tsx`, `cscreen__topbar` header, ~lines 128–170):
- This screen has no wallet today and gains a new `transcendShards: number` prop.
- Render the shard chip in `.cscreen__topbar`, **right-aligned, before the `.cscreen__close` button** (mirrors where the gacha wallet sits relative to its close button). Same glyph/color/size as the gacha instance.
- Keep it out of the roster scroll region so it stays pinned while the roster scrolls.

**Consistency requirement:** the two instances must be visually identical (same glyph, color, size, format, aria-label pattern). Same component/class in both.

## Copywriting Contract

This phase is a **read-only balance readout** — no CTA, no form, no error path, no destructive action. The only user-facing copy is the accessible label.

| Element | Copy |
|---------|------|
| Primary CTA | none (no action in this phase) |
| Empty state heading | none — `0` renders in the chip; no empty screen exists |
| Empty state body | none |
| Error state | none — value is a subscribed integer that always resolves via `?? 0`; there is no failure surface to message |
| Destructive confirmation | none — no destructive action in this phase |
| Shard chip accessible label (LV) | `Zvaigžņu šķembas: {n}` |

Non-blocking naming note: the code identifier is `transcendShards`; the chosen Latvian user-facing term is **"Zvaigžņu šķembas"** (constellation/star shards), consistent with the existing `ZVAIGZNES` constellation labelling. If the planner prefers "Transcendences šķembas", swap the label string only — glyph/placement/format are unaffected.

## Interaction & States

| State | Behaviour |
|-------|-----------|
| Default | Chip shows `◈ {n}` in gold; static, non-interactive (display only). |
| Zero | Shows `◈ 0`; no dimming, no hint text, always visible. |
| Update (mint) | Value re-renders reactively when the subscribed `player.transcendShards` increments (SpacetimeDB subscription push). No animation required in this phase — per-pull "+1 shard" feedback is explicitly **deferred to Phase 2** (see RESEARCH open question 1). |
| Hover/focus | Inherit existing wallet behaviour (no special affordance; the chip is not a button). |
