<objective>
Add the server-authoritative party model and its five reducers to the SpacetimeDB module: the
`party` / `party_member` / `party_invite` tables plus `invitePlayer`, `requestJoin`,
`acceptInvite`, `declineInvite`, `leaveParty`. All grouping state, authz, cap/uniqueness
enforcement, leader promotion, disband, and state-based invite invalidation live here.

Purpose: STDB is server-authoritative — uniqueness (one party per player), the RAID_PARTY_SIZE
cap, promotion (D-05), and invite invalidation (D-08) must be enforced in reducers or they can be
bypassed. The security-critical accept path stays symmetric across invite/ask-to-join by modelling
joiner/recipient explicitly (never branching authz on `kind`).
Output: additive schema + reducers compiled clean; rule logic delegated to Plan 01's pure helpers.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → reducer args | Untrusted `targetIdentity` / `inviteId` cross here. The actor is NEVER a client arg — always `ctx.sender → accountIdentity(ctx)`. |
| reducer → tables | All party state mutation is server-authoritative; the DB `unique()` constraint is the atomic race-free backstop. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-05-01 | Spoofing | all 5 reducers | high | mitigate | Actor resolved via requirePlayer(ctx)/accountIdentity(ctx); passed identities are lookup targets only (critical rule 5). |
| T-05-02 | Elevation of Privilege | acceptInvite / declineInvite | high | mitigate | Guard `inv.recipientIdentity.equals(me.identity)` before any membership change — only the recipient may accept/decline. |
| T-05-03 | Tampering | membership insertion | high | mitigate | No joinParty(rawId) reducer; party_member rows are only ever inserted from a matching pending party_invite. |
| T-05-04 | Tampering | acceptInvite cap/uniqueness | high | mitigate | canAccept (Plan 01) enforces cap + no-double-join at ACCEPT time; party_member.identity.unique() is the atomic backstop against concurrent accepts. |
| T-05-06 | Denial of Service | invite spam | low | accept | Dedupe pending invites per (partyId, joinerIdentity); LAN scale is tiny. Accepted residual risk. |
| T-05-SC | Tampering | package installs | low | accept | Zero package installs this phase. |
</threat_model>
